## 1. 核心洞察：最大槓桿 = 共用「生物力學幾何引擎」

目前痛點本質：**只有「分類/事件」，沒有「可解釋的數字」**——ONNX 說 over the top 但說不出差幾度；節奏有 ±11% 量化漂移；P-System 只有 8 個時間點、無任何角度。

一個純裝置端、純幾何的 **`BiomechanicsService`**（從既有 33 點 pose CSV 算 X-factor / 脊椎前傾 / 肩髖旋轉 / 重心轉移 / 頭部位移 / wrist hinge）**一次餵 5 個子系統**：P1-P10、骨架姿勢量化、AI 建議、修正追蹤、節奏速度。

### 跨子系統 enabler（共用一次、多處受益）
- **肩寬正規化 + 歸一化座標統一**：取代所有 `px/frame` 寫死門檻（解析度/機位相依）
- **`hitSecond` 統一成「相對擊球秒 Δt」**：擊球判定/切片邊界/光暈/疊影對齊/桿頭觸球共用同一觸球真值
- **共用疊圖元件**（PoseTrack+SkeletonPainter+TrajectoryPainter+新 GhostSkeletonPainter+PoseNormalizer）
- **`phases.json` 8→10 schema**：P-System 單一時間真相來源
- **音訊 onset 偵測升級**：同時改善觸球時刻定位 + 甜蜜點評分
- **真實 fps + `ptsUs` 時間擬合**：球軌跡/節奏/桿頭速度共同地基（目前多處假設 30fps 寫死）

## 6. Wave 2 深掘：BiomechanicsService

### 資料模型
- `BiomechMetric{key, value, ideal(min,max), deviation, grade}`
- `PSystemMetrics{pSec(P1..P10), perP(每P指標), xFactorSeries, tempoRatio}` → 寫 angles.json（補角度不持久化缺口）

### 指標與公式（MediaPipe 索引 0 鼻/11-12 肩/13-14 肘/15-16 腕/23-24 髖/27-28 踝）
| 指標 | 公式 | 取樣時點 |
|------|------|----------|
| 肩旋轉角 | 11→12 連線相對 address 基準角 | 全程，P4 峰值 |
| 髖旋轉角 | 23→24 同理 | 同上 |
| X-Factor | 肩旋轉−髖旋轉；轉移序列由兩角速度過零先後判髖先於肩 | P4 峰值+全程 |
| 脊椎前傾角 | 髖中點→肩中點向量 vs 垂直線夾角 | P1 基準、P7 比較 |
| 頭部位移 | 0 相對 address 歸一化位移框 | 全程 max sway/lift |
| 重心轉移 | 髖中點 x 相對雙踝中點左右比例 | backswing→impact |
| Wrist hinge | 腕15-肘13-肩 三點夾角 | downswing 時序 |

### 取樣時點
直接吃 SwingHit 的 P1-P10 秒數（Wave 1 修穩後 phases.json 8→10），不自行重算階段。

### 視角閘門
用肩寬/髖寬比例粗判視角，face-on 才顯示旋轉角，否則標「需側面拍攝」。全程標「個人相對指標、非實驗室級」。

### 整合點（一 service，四處消費）
1. P1-P10：perP 給每 P 角度+評分（規則表：P4 肩旋轉 80-95°、X-factor 35-55°）
2. 骨架量化卡 MetricsBreakdownCard：數值+理想區間 bar+差 X 度
3. AI 建議：序列化成 swing_metrics JSON 注入 Gemini prompt，evidence 必須引用具體角度
4. 修正追蹤：寫回 entry→個人 baseline/z-score→practiceSuggestion→targetMetric 自動驗證閉環

### 落地順序
抽共用幾何工具（向量夾角/atan2/肩寬正規化，抄 swing_stats_service）→ 6-7 指標 per-frame 計算+視角閘門 → angles.json 持久化 → MetricsBreakdownCard → 後端 SwingMetricsService.cs 鏡像（**Dart↔C# 數值一致性測試必做**，否則即時與雲端結論打架）

### 最大風險
Dart（即時）與 C#（Gemini 注入）兩端各算角度，normalize/視角處理必須一致，需共用「黃金測試向量」做三方回歸。
