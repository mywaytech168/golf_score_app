# 錄影歷史本地/雲端分類系統設計

## 概述

本文檔描述了新實施的錄影歷史分類系統，將使用者的錄影分為「本地」和「雲端」兩個分類，支援上傳狀態追蹤、自動同步和智能重試。

## 核心架構

### 1. 數據模型升級（RecordingHistoryEntry）

#### 新增字段

```dart
/// 上傳狀態枚舉
enum UploadStatus {
  local,      // 本地只 - 未上傳
  uploading,  // 上傳中
  uploaded,   // 已上傳
  failed      // 上傳失敗
}

class RecordingHistoryEntry {
  // 原有字段...
  
  // 新增上傳追蹤字段
  final UploadStatus uploadStatus;        // 上傳狀態
  final int? cloudVideoId;                // 雲端視頻 ID（已上傳時有值）
  final String? uploadError;              // 失敗原因
  final DateTime? lastUploadAttempt;      // 最後上傳嘗試時間
}
```

#### JSON 序列化/反序列化

已更新 `toJson()` 和 `fromJson()` 方法，支援新字段的持久化：

```dart
// 保存時自動包含上傳狀態
Map<String, dynamic> toJson() {
  return {
    // ...原有字段
    'uploadStatus': uploadStatus.name,
    'cloudVideoId': cloudVideoId,
    'uploadError': uploadError,
    'lastUploadAttempt': lastUploadAttempt?.toIso8601String(),
  };
}

// 載入時自動恢復，預設為 local
factory RecordingHistoryEntry.fromJson(Map<String, dynamic> json) {
  final statusStr = json['uploadStatus'] as String?;
  UploadStatus uploadStatus = UploadStatus.local;
  if (statusStr != null) {
    uploadStatus = UploadStatus.values.byName(statusStr);
  }
  // ...
}
```

### 3. 上傳管理服務（RecordingUploadManager）

**文件**: `lib/services/recording_upload_manager.dart`

#### 職責

- 管理錄影的上傳狀態
- 自動重試失敗的上傳
- 提供進度回呼
- 通知 UI 狀態變化（ChangeNotifier）

#### 主要方法

```dart
class RecordingUploadManager extends ChangeNotifier {
  // 初始化，從本地存儲讀取狀態
  Future<void> initialize() async { ... }

  // 上傳單個錄影
  Future<int?> uploadRecording(
    RecordingHistoryEntry entry,
    {ValueChanged<double>? onProgress}
  ) async { ... }

  // 批量上傳所有本地錄影
  Future<int> syncAllLocal() async { ... }

  // 重試失敗的上傳
  Future<int> retryFailedUploads() async { ... }

  // 刪除本地錄影
  Future<void> deleteRecording(RecordingHistoryEntry entry) async { ... }

  // 刪除雲端錄影（本地副本）
  Future<void> deleteCloudRecording(RecordingHistoryEntry entry) async { ... }

  // 取得上傳狀態和進度
  UploadStatus? getUploadStatus(String filePath);
  double getUploadProgress(String filePath);
}
```

## 工作流程

### 用户场景 1：上傳單個本地錄影

1. 使用者在「本地」標籤頁看到未上傳的錄影
2. 點擊「上傳」按鈕
3. 系統調用 `RecordingUploadManager.uploadRecording()`
4. 錄影狀態變為 `uploading`，顯示進度條
5. 上傳完成後：
   - 狀態變為 `uploaded`
   - `cloudVideoId` 被設置為服務器返回的 ID
   - 記錄保存到本地 JSON
6. 使用者可在「雲端」標籤頁看到已上傳的錄影

### 用户场景 2：批量同步所有本地錄影

1. 使用者在「本地」標籤頁點擊「全部上傳」按鈕
2. 系統調用 `RecordingUploadManager.syncAllLocal()`
3. 逐個上傳所有本地錄影，顯示整體進度
4. 完成後顯示成功/失敗統計

### 用户场景 3：重試失敗的上傳

1. 使用者在「本地」標籤頁看到狀態為 `failed` 的錄影
2. 點擊「重試」按鈕
3. 系統清除之前的 `uploadError`，重新嘗試上傳
4. 成功則轉移到「雲端」標籤頁

### 用户场景 4：刪除錄影

- **本地刪除**: 從本地 JSON 移除，同時刪除實體檔案
- **雲端刪除**: 調用後端 API 刪除服務器副本，然後同步本地

## 关键设计决策

### 1. 為什麼使用狀態機而非布林值？

使用 `UploadStatus` 枚舉而非簡單的 `isUploaded` 布林：
- **優點**: 更清晰地表達複雜的上傳流程（上傳中、失敗、重試）
- **擴展性**: 未來容易添加新狀態（如 syncing、paused）

### 2. 為什麼保留本地副本？

即使上傳成功，也在本地保留錄影文件：
- **離線訪問**: 用戶可離線查看和分享錄影
- **備份**: 雲端作為備份，本地作為主副本
- **靈活刪除**: 用戶可選擇只刪除本地或同時刪除雲端

### 3. 為什麼自動重試？

上傳失敗時自動進入重試隊列：
- **用戶友好**: 無需用戶手動干預，網絡恢復後自動重試
- **可靠性**: 提高最終上傳成功率

## 常見問題

**Q: 如果上傳失敗，數據會丟失嗎？**
A: 不會。錄影文件和元數據都保存在本地，用戶可隨時重試上傳。

**Q: 同時編輯和上傳會怎樣？**
A: 系統使用樂觀鎖定（版本號），確保數據一致性。如衝突，以最新修改為準。

**Q: 如何處理大文件上傳？**
A: 實施分塊上傳（chunked upload）和斷點續傳（resume），見後端實現待補充部分。

**Q: 能否自定義上傳時機？**
A: 可以。用戶可選擇手動上傳或設置 WiFi 時自動上傳（需添加功能）。
